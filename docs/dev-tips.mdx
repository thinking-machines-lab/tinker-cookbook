# Developer Tips

## Error Handling and Retries

The Tinker SDK includes built-in retry logic for all HTTP API calls (training, sampling, checkpointing, and service operations). You do **not** need to add your own retry wrappers around Tinker API calls.

### What gets retried automatically

Every API request made through the Tinker SDK goes through a unified retry mechanism that handles:

- **Timeouts** (HTTP 408) and connection-level timeouts
- **Lock conflicts** (HTTP 409)
- **Rate limits** (HTTP 429)
- **Server errors** (HTTP 500+)
- **Connection failures** (network interruptions, DNS errors, etc.)

The SDK also respects the server's `Retry-After` header when present, and attaches idempotency keys to non-GET requests so that retried writes are safe.

### Retry configuration

| Parameter | Default | Description |
|---|---|---|
| Max retries | 10 | Maximum number of retry attempts per request |
| Initial delay | 0.5 seconds | Starting backoff interval |
| Max delay | 10 seconds | Cap on backoff interval |
| Backoff strategy | Exponential with jitter | `min(0.5 * 2^attempt, 10) * jitter` |

You can adjust the retry count when creating a client:

```python
# Reduce retries (e.g. for fail-fast scripting)
service_client = tinker.ServiceClient(max_retries=3)

# Disable retries entirely
service_client = tinker.ServiceClient(max_retries=0)
```

### What does NOT get retried

Client-side errors (4xx other than 408, 409, 429) are **not** retried because they indicate a problem with the request itself:

- **400 Bad Request** — malformed data, invalid parameters
- **401 Unauthorized** — missing or invalid API key
- **403 Forbidden** — insufficient permissions
- **404 Not Found** — invalid model path or resource
- **422 Unprocessable Entity** — semantically invalid request

These raise the corresponding exception (`tinker.BadRequestError`, `tinker.AuthenticationError`, etc.) immediately.

### Logging retries

The SDK logs retry activity through Python's standard `logging` module under the `tinker` logger. To see retry-related messages during development:

```python
import logging
logging.getLogger("tinker").setLevel(logging.DEBUG)
```

### Implications for tinker-cookbook

Since the SDK handles retries internally, `tinker-cookbook` training loops (SL, RL, DPO, distillation) do **not** wrap API calls with additional retry logic. If you are building your own training loop on top of `tinker-cookbook` utilities, you generally do not need to add retry handling for Tinker API calls — just handle the final exception if all retries are exhausted.

For non-Tinker external services (e.g., third-party embedding APIs, vector databases), you should implement your own retry logic as appropriate. See `tinker_cookbook/recipes/search_tool/` for an example.
